Who They Are

Multi-Brand Manufacturer, Bristol Indiana — 137 Dealers Across the US

Viaggio Pontoons is the lead brand in a portfolio of four manufactured goods companies operating out of Bristol, Indiana — alongside Lakota Trailers, SMC, and Duo Form. Nathaniel Talbot, Director of IT and ERP Solutions Manager, is a team of one managing a 15-year career in enterprise IT and a vision that runs well beyond a single brand: build a platform once for Viaggio, then extend it to Lakota, SMC, and Duo Form without rebuilding middleware. Their current platform, NetSuite SuiteCommerce (~$35K/yr), was explicitly described as "never designed for B2B" by NetSuite's own engineers. Simple UI changes take 6+ months through third-party developers. The goal is a unified Shopify B2B dealer portal and D2C boat owner portal on a single platform — with a rule-based boat configurator, lot inventory management, and two-phase dealer registration — timed to the Model Year 2028 launch.

Shopify Delivers

Eight requirements from Viaggio's RFI where Shopify Plus meets or exceeds expectations — natively or with a clear, well-scoped build pattern.

  • 1
    Section 9.2 · B2B Dealer Portal
    137 Dealers, Multi-Tenant Isolation — Native
    Each dealer is a Company in Shopify B2B — with their own Company Location, Contacts, price list, and payment terms. Data isolation between dealers is enforced at the platform level. A dealer logged in cannot see another dealer's orders, inventory, or registration data. Shopify Plus B2B Company Accounts are designed for exactly this architecture. No custom multi-tenancy code required.
  • 2
    Section 9.1 · Dealer Configurator
    Rule-Based Configurator on Hydrogen + Shopify Infrastructure
    Shopify doesn't sell a boat configurator. What it provides is the platform to build one correctly: Storefront API for commerce data, Shopify Functions for business logic evaluation, Metaobjects as the structured rule store (editable by Viaggio admins without a code deployment), and Hydrogen as the React framework. The configurator lives as a headless application on Vercel — independent from the storefront, embeddable anywhere, and not constrained by Shopify's checkout flow. This is the architecture that meets every configurator requirement in the RFI.
  • 3
    Section 9.1.4 · Dual Pricing
    MSRP / Dealer Cost Dual Pricing — Native, No Custom Code
    An unauthenticated visitor to the public configurator sees MSRP. A dealer who logs into their B2B portal session sees their dealer cost. Shopify B2B price lists respond to the authenticated session context automatically — the same configurator serves both audiences. This is a core feature of Shopify B2B, not a workaround.
  • 4
    Section 9.1.5 · WordPress Embed
    Configurator Embeds in WordPress — WordPress Stays Unchanged
    Because the configurator is a headless application on Vercel, it is completely independent of Shopify's storefront layer. It can be embedded in any web context — including the current WordPress site — via iframe or a lightweight JavaScript initialization script that passes model ID, user role, and authentication context. WordPress does not need to be migrated or rebuilt. This is an architectural property of headless, not a feature request.
  • 5
    Section 9.3 · NetSuite Integration
    Integration Pattern Nathaniel's Team Already Knows
    Shopify does not call NetSuite directly — and that's by design. A Vercel serverless function layer sits between Shopify webhooks (order created, registration submitted, boat shipped) and NetSuite RESTlets. NetSuite credentials live in Vercel environment variables, never visible in the browser. Nathaniel's team has RESTlet expertise — they are well-positioned to own or co-develop this layer. The pattern is familiar, the stack is standard, and no Shopify constraint prevents any of the integration flows described in the RFI.
  • 6
    Section 9.2.4 · Lot Inventory
    Hull Records as Metaobjects — Not a Product Catalog
    Hull IDs are not products. They're records — with a hull ID, model, year, options, status, dealer assignment, and lot entry date. Shopify Metaobjects model this precisely: Admin API creates a hull record when NetSuite fires a "boat shipped" webhook, updates it when the registration is submitted, and marks it "sold" when a retail registration completes. Dealers see their live lot via the portal. Nothing gets forced into a product catalog paradigm it doesn't fit. The architecture is purpose-built for Viaggio's lot management workflow.
  • 7
    Section 9.4 · Mobile PWA
    Installable PWA — No App Store, Camera Access, Offline Queue
    A Hydrogen + Next.js application on Vercel supports full PWA capability: installable to home screen on iOS and Android with no App Store submission, camera access via the browser MediaDevices API (bill of sale photograph, hull ID QR scan, delivery photos), and an offline service worker + IndexedDB queue for registration submissions at boat shows or dealer lots with poor connectivity. These are standard web platform capabilities — not Shopify-specific features — and they work correctly in the recommended headless stack.
  • 8
    Nathaniel's Explicit Requirement
    No IP Lock-In. Current Tech Standards. Architecture Replicates to All Four Brands.
    Everything Shopify stores is accessible via Admin API — products, orders, customers, Metaobjects, company accounts — exportable at any time in standard formats. The configurator is a React/Next.js application, no proprietary framework. Rules are stored as JSON in Metaobjects. If Viaggio migrates off Shopify, they keep the configurator and the rules. For multi-brand replication: one Shopify Organization, expansion stores per brand (Viaggio → Lakota → SMC → Duo Form), shared middleware. The architecture Nathaniel designs for Viaggio is, by default, the architecture he extends to the other three brands without rebuilding anything.

What requires custom development: The configurator UI, multi-step delivery registration workflow, lot inventory interface, e-signature integration, two-phase registration state machine, offline service worker, and push notification layer are all custom builds on top of Shopify's platform infrastructure. This is expected for a project of this scope. The platform provides the commerce layer, data layer, and API surface — the applications that run on top are purpose-built for Viaggio.

Where Shopify Wins Outright
  • Architecture flexibility — headless, composable, embeddable. SuiteCommerce is a monolith.
  • Developer ecosystem — hiring for React/Next.js is a global talent pool. Hiring for SuiteCommerce is a niche market.
  • Multi-brand replication — Shopify Organization was designed for exactly what Nathaniel described. SuiteCommerce was not.
  • Speed of change — Nathaniel can update Shopify admin settings in minutes. SuiteCommerce changes require third-party engagement.
Where NetSuite Has an Edge
  • Native ERP connection — same vendor, same system. No middleware required for basic catalog and order sync. Shopify closes this gap with the Vercel integration layer, but it is a build.
  • Existing relationship — Viaggio is already on NetSuite for ERP. The SuiteCommerce path would be a lighter integration. Shopify's path is more powerful but requires more integration design work upfront.

Nathaniel's own 5-year cost analysis concluded that Shopify is the right platform. This comparison reinforces why.
